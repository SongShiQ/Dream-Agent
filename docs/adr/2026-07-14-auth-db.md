# ADR 2026-07-14: 身份、权限与数据库生产前提

**状态**：建议采纳，待负责人确认  
**适用范围**：导学、基础 OJ、专业 OJ、项目候选与运营后台  
**关联计划**：`docs/plans/2026-07-14-funnel-oj-detailed-work-plan.md`

## 背景

训练营入口约 2,000 人，后续基础、专业和项目阶段都会产生可影响晋级的证据。当前实现仍存在客户端传入 `studentId`、开发库使用 SQLite、OJ schema 缺 migration 等问题。若直接进入真实 OJ，会把身份伪造、跨端进度丢失和多 worker 写入冲突带进核心链路。

## 决策

1. 正式环境使用 PostgreSQL；SQLite 仅作为本地开发和轻量测试数据库。
2. 所有写入型 API 必须从服务端 session 推导当前用户与 student，不接受浏览器传入的 `studentId` 作为资源所有权依据。
3. 角色分为 `student`、`ta`、`teacher`、`admin`。默认最小权限：学生只能读写自己的学习与提交；TA 可读风险队列与分配学生；教师可配置课程规则和复核项目候选；admin 负责系统配置与审计。
4. `Student` 必须绑定 `cohortId`；`cohort` 必须绑定 `curriculumVersion`。题库、gate、判题镜像和晋级规则都按 curriculum version 解释。
5. 课程证据分为 `viewed`、`personal_done`、`mastered`。只有服务端评分、小测达标、工具链自检达标和 OJ AC 可形成 `mastered`；个人勾选不得晋级。

## 推荐默认值

| 项目 | 默认值 |
|---|---|
| 登录方式 | 学号/邮箱 + 一次性验证码或统一身份登录；本地开发保留显式 dev user |
| session | HttpOnly cookie；服务端 route 统一读取；禁止前端自行声明角色 |
| 生产数据库 | PostgreSQL，优先 Supabase 或等价托管 Postgres |
| 本地数据库 | SQLite，可通过 seed 创建 dev student |
| 审计 | 记录身份、角色、资源、动作、结果、请求 id；不记录敏感 token |
| 迁移 | 只新增 migration，不修改历史 migration；迁移失败时进入维护模式 |

## API 边界

| API 类别 | 身份规则 | 权限规则 |
|---|---|---|
| `/api/me/*` | 必须登录 | 只能访问当前 session 绑定 student |
| `/api/assess` | 必须登录 | 学生写自己的答案；TA/teacher 只读聚合 |
| `/api/submit` | 必须登录 | 学生提交自己的 gate；gate 解锁由服务端判断 |
| `/api/labs` | 必须登录 | 学生只读自己的 gate 状态；不得写 passed |
| `/internal/*` | 不使用学生 session | 仅 worker/service token；不暴露给浏览器 |
| `/api/admin/*` | 必须登录 | 仅 teacher/admin，全部写审计 |

## 影响

- M1-A 必须先补齐 Prisma migration，并验证空库和开发库。
- M1-B 必须先完成 session-derived identity，再改 dashboard、assess、plan、labs、submit。
- 生产上不能把 SQLite 作为多 worker 写库方案，也不能让前端 `studentId` 决定读取或写入对象。

## 未决问题

1. 正式登录系统使用学校统一身份、邮箱验证码，还是训练营自建账号。
2. PostgreSQL 托管方和备份责任人。
3. TA 分配机制：按 cohort、按阶段，还是按风险队列自动分配。

## 验收

- 未登录写请求返回 401；越权访问返回 403。
- A 学员无法读取或修改 B 学员的计划、答案、提交和 gate。
- `studentId` 出现在请求体时只作为兼容字段记录/忽略，不作为授权依据。
- PostgreSQL migration 可重复部署；SQLite 仅用于本地开发。
