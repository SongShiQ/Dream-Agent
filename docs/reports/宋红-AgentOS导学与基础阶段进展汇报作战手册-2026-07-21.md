## 三、四个开源项目到底是什么

### 1. HKUDS/DeepTutor

项目地址：[HKUDS/DeepTutor](https://github.com/HKUDS/DeepTutor)  
本地审计提交：`3e3https://github.com/HKUDS/DeepTutorb9a6`

DeepTutor 是完整的 AI Tutor 运行时，不只是聊天机器人。它覆盖 chat、复杂解题、研究、可视化、知识库、用户记忆和 mastery path。最值得吸收的不是界面，而是这条原则：
*界面类似于Gemini web端*
> LLM 可以决定“怎么解释、怎么引导”，但学生是否掌握、是否可以前进，应由确定性 policy、gate 和 scheduler 决定。

能直接用的部分：

- 可以独立部署做 Tutor 和 mastery 体验 POC。
- 可以连接 Markdown/Obsidian Wiki，快速验证“知识库 + Tutor”体验。

不直接整仓接入的原因：

- Dream Agent 已有学生入口、会话、学习状态和证据库。
- 整仓接入会形成两套 UI、两套用户状态、两套 mastery 和更复杂的数据同步。
- DeepTutor 的本地执行能力不能替代多租户 OS Judge。

当前吸收结果：

- mastery 只接受可信证据，不接受 LLM 自报。
- Dashboard 根据学习状态给唯一下一任务。
- high-stakes 失败后安排补弱和 alternate set。
- ReviewSchedule 做确定性间隔复习。
- 复习只更新复习调度，不替代首次 mastery 证据。

### 2. ishicm/llm-wiki-skills

项目地址：[ishicm/llm-wiki-skills](https://github.com/ishicm/llm-wiki-skills)  
本地审计提交：`2047d5e`

它不是知识库服务，而是一套面向 Agent 的 Markdown Wiki 操作规程，主要定义：原始资料如何保留、Wiki 如何组织、索引如何同步、重构何时需要确认、如何维护来源和结构。

能直接用的部分：

- 直接吸收内容治理规则和工作纪律。
- 用作知识卡 schema、稳定 ID、来源引用、审核和发布检查的设计种子。

不能直接承担的部分：

- 没有稳定解析器、API、**数据库、锁、多用户权限**和自动化测试。
- “支持 PDF/URL”取决于承载它的 Agent 是否有相关工具，不是仓库自身提供生产流水线。

当前吸收结果：

- 知识卡稳定 ID 和 `[K:<id>]` 引用。
- 来源、审核状态、reviewer、reviewed_at 和内容 hash。
- `content:release-check` 的 lint、warning、blocker 和 release 模式。
- 教师决策不直接改内容文件，而是经过 hash 检查和原子 apply。

### 3. VectifyAI/OpenKB

项目地址：[VectifyAI/OpenKB](https://github.com/VectifyAI/OpenKB)  
本地审计提交：`e6f3285`

OpenKB 是 Python CLI 知识编译器，不是**多租户在线知识服务**。它可以摄入 PDF、Word、PPT、Excel、HTML 和 URL，将资料编译成 summary、concept、entity、source、index 和 log；长 PDF 可以使用 PageIndex 树索引，还提供 query、chat、lint、visualize、Skill 和 deck 生成。

能直接用的部分：

- 固定版本后，作为教师或 CI 侧离线工具直接运行。
- 复用其多格式解析、Wiki 编译、结构 lint、原子写和崩溃安全能力。

不能直接放到学生请求热路径的原因：

- 当前仍是 Alpha CLI/本地文件系统产品，不是课程级多租户服务。
- 学生每次请求触发 Python CLI 和额外 LLM 编译，会增加延迟、成本和内容即时漂移风险。
- OpenCamp 还需要**课程版本、审核状态、稳定 ID 和发布权限**，这些不是 OpenKB 原生职责。

当前吸收结果：

- `lib/knowledge/openkb-import.ts` 做 OpenKB 到 Dream Agent 的单向导入适配。
- manifest/source contract 进入统一发布检查。
- OpenKB 仍位于教师/CI 离线侧，学生只读发布后的 Git 内容包。

### 4. nashsu/llm_wiki

项目地址：[nashsu/llm_wiki](https://github.com/nashsu/llm_wiki)  
本地审计提交：`969e7e8`

它是完整的 Tauri v2 + Rust + React 桌面知识产品，不是轻量脚本。已有多格式摄入、网页剪藏、Wikilink 图谱、混合检索、审核队列、Deep Research、本地 HTTP API 和 MCP Server。

能直接用的部分：

- Windows portable/EXE/MSI 可直接作为学生私人书房，也可用于教师资料摄入、图谱和问答演示。
- 可通过导出、HTTP API 或 MCP 与 Dream Agent 松耦合。

不并入核心服务的原因：

- 它是本机单用户桌面应用，不能承担在线课程 SLA。
- GPL-3.0 源码直接合并需要评估组合分发义务。
- 本地审计发现根 `package-lock.json` 与 `package.json` 不同步，`npm ci` 实测失败。

当前吸收结果：

- 定位为学生可选的“私人书房”，默认私有，不直接写入公共课程知识或 mastery。
- 不复制 GPL-3.0 Rust/TS 源码进入 Dream Agent。
- 等出现真实使用需求后，再通过导出/API/MCP 与 Dream Agent 松耦合。

### 5. 四仓横向结论

| 项目              | 最成熟能力                          | 当前采用方式                  | 是否直接进入学生线上主链路 |
| --------------- | ------------------------------ | ----------------------- | ------------- |
| DeepTutor       | Tutor、mastery、policy、scheduler | 移植教学状态思想；可独立 POC        | 否             |
| llm-wiki-skills | Wiki 维护规程                      | 直接转成治理规则和发布检查           | 否             |
| OpenKB          | 多格式摄入和 Wiki 编译                 | 教师/CI 离线 sidecar + 单向导入 | 否             |
| nashsu/llm_wiki | 个人桌面知识管理                       | 学生可选私人书房；导出/API/MCP 松耦合 | 否             |
| Dream Agent     | 学生入口、学习状态、证据、Dashboard         | 继续作为唯一学生系统              | 是             |

这四个项目都不提供正式 OS/QEMU 隔离判题和可信 AC，因此实验 Judge 最终仍需要自研；但按当前重点，这部分暂缓。

## 五、当前导学与基础闭环

```text
学生进入系统
  -> Dashboard 读取学习状态
  -> 给出唯一下一任务
  -> 阅读教师知识卡 / Tutor 使用 [K:id] 引用
  -> 启动本单元 Foundation 小测
  -> 服务端检查前置单元、当日高 stakes 次数和题量
  -> 按标签与难度蓝图选择 5 题
  -> 提交后服务端评分
      -> 通过：形成 mastery 证据，创建后续复习计划
      -> 失败：不授予 mastery，归因薄弱知识点，推荐知识卡
  -> ReviewSchedule 到期后进入 practice
  -> 刷新页面仍从 AnswerRecord 重建最近诊断
  -> Dashboard 重新计算下一任务
```

### 1. 导学阶段已完成

- Dream Agent 是唯一学生入口。
- Tutor 答案可以携带稳定 `[K:<id>]` 引用，引用会随 Chat 持久化。
- Dashboard 根据 assessment、Foundation、复习和 Gate 状态选择唯一主任务。
- Foundation 前置关系在服务端校验，不能通过手工 API 请求绕过前端锁定。
- 失败后按知识点返回 `weakPoints`、最多 3 张补弱知识卡和确定性 nextAction。
- 最近一次诊断从 AnswerRecord 重建，刷新 Foundation 页面或主 Dashboard 不丢失。
- 空 weakPoints 不再错误推荐 Rust；Foundation 已失败/进行中的 OS 单元优先保持为主任务。
- 取消复习后恢复上一条诊断，不残留误导性“复习已开始”提示。
- 390×844 移动端标题、知识卡、诊断和按钮均无横向溢出。

### 2. 基础阶段已完成

- 6 个必修 Foundation 微单元：OS 总览与中断、进程与调度、内存与虚存、Rust 基础、所有权与错误处理、工具链与读代码。
- 每次小测 5 题，达标线 80%。
- high-stakes 每日 1 次；首次失败后要求使用 alternate set。
- 每个单元至少 10 道合格题，确保首轮与失败后重测不重复。
- 选题不再从其他主题补题，也排除 professional 阶段题目。

### 1. 这四个项目到底有什么区别？

> DeepTutor 管“怎么教和怎么推进”，llm-wiki-skills 管“Agent 按什么规矩维护知识”，OpenKB 管教师公共藏书阁，nashsu/llm_wiki 管学生私人书房。Dream Agent 管统一学习入口和可信学习证据。

> 四个都能在各自边界直接用：DeepTutor 做独立 Tutor POC，OpenKB 做教师侧离线编译，llm-wiki-skills 做规程，llm_wiki 用 Windows 安装包做学生个人知识管理。但没有一个能单独成为我们的 OS 自学系统。

### 3. 为什么不直接部署 DeepTutor？

> 会形成两套学生账户、会话和 mastery。当前最有价值的是移植其确定性推进机制，Dream Agent 继续持有唯一学习状态。

### 4. OpenKB 现在是否已经在线接入？

> 没有，也不计划放学生热路径。当前是单向导入、manifest/source 检查和发布门禁。学生读取的是已经进入 Git 内容包的稳定知识。

### 5. llm-wiki-skills 不就是一份 Prompt 吗？

> 是操作规程，不是引擎。它的价值在来源不可丢、索引同步、结构变更确认和定期 lint；我们已经把这些原则落实成 schema、hash、审核和 release check。

### 6. 为什么不把 nashsu/llm_wiki 源码搬过来？

> 它是 GPL-3.0 的单机桌面产品。当前最稳妥的方式是独立安装，通过导出/API/MCP 松耦合，不让在线学生系统依赖某台教师电脑。

### 7. 你现在是不是已经有完整 AI Tutor？

> 还不是。当前完成的是可信引用、确定性学习状态、Foundation 证据、复习和下一任务。复杂的生成式个性化教案和大规模效果验证还没有完成。

### 8. Mastery 是怎样获得的？

> Foundation mastery 只接受服务端 high-stakes quiz，当前每次 5 题、80% 通过。Tutor 对话、知识浏览、practice 和形成性实验都不能直接授予 mastery。

### 9. 为什么每个单元需要 10 题？

> 首轮 high-stakes 使用 5 题，首次失败后策略要求 alternate set。如果只有 5 题，重测会重复，因此发布门槛是两套各 5 题。

### 10. 标签覆盖为 0 是否说明题库质量完成了？

> 只能说明六个单元的目标细标签都有至少一道题，题量和标签门槛已经满足。它不等于教师认可题目质量，也不等于错误归因对真实学生足够有效，下一步仍要做主题包抽检和学生试点。

### 11. 48 个 blocker 为什么不直接修掉？

> 因为它们主要是教师审核和发布决策。工程可以修格式、来源和状态机，但不能替老师判断知识内容是否正确，也不能伪造 reviewer 和 reviewed_at。

### 12. 现在学生会不会看到未经审核的知识？

> 当前数据存在 `published_pending`，release check 已将其作为 blocker。是否立即在运行时收紧为 reviewed-only 需要老师拍板，这是本次最重要的发布决策之一。

### 13. 测试通过能否证明学习效果？

> 不能。测试证明工程行为可重复。学习效果需要真实学生的首次通过率、补弱后提升、完成时间和主观反馈。

### 14. 现在最值得演示什么？

> 演示一个学生从 OS 总览小测失败开始，看到薄弱点和 `[K:id]` 卡片，刷新后诊断仍在，启动到期复习，取消后恢复旧诊断，Dashboard 继续推荐同一个 OS 单元。

### 15. 当前最大的工程风险是什么？

> 不是模型调用，而是内容版本、审核状态、题目标签和学习证据之间的漂移。当前通过稳定 ID、hash、release check、服务端选题和 AnswerRecord 重建控制风险。

### 16. 为什么实验阶段暂缓仍保留 Gate 并发修复？

> Gate 是学生 Dashboard 和当前基础路径已有的共享状态。并发初始化 500 会直接影响前两个阶段，因此修复属于主链路稳定性，不代表重新启动实验工程。

### 17. 下一周期只能做一件事，选什么？

> 选“内存与虚存”第三主题包，但先做证据审计。前两个主题已经证明契约可复用，第三个主题可以补齐连续 OS 导学链路；如果真实题目、补弱卡或前置关系不足，就不创建空包，转而优先做题目质量抽检。

### 18. 希望老师这次给什么结论？

> 请确认 reviewed-only 的发布口径、两个已完成主题包的内容表述、5 题 80% 的 mastery 标准，以及是否基于当前双主题 5/5 工程样板冻结一次工程基线。
