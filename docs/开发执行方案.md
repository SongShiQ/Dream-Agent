## 二、开发顺序

### 核心原则

1. **先基础后上层**：数据库 → API → 前端
2. **先核心后扩展**：知识库 → 记忆系统 → AI 分析
3. **先可用后完善**：基本功能 → 高级功能 → 优化

### 推荐顺序

```
阶段 1: 数据基础 (1-2 天)
├── 数据库迁移
├── 基础 API (chat, assess, exam, plan, submit)
└── 学员管理 API

阶段 2: 知识基础 (2-3 天)
├── 知识库构建
├── 题库导入
└── 知识检索 API

阶段 3: 记忆系统 (2-3 天)
├── L1 原始追踪
├── L2 表面摘要
├── L3 跨表面综合
└── 知识点管理

阶段 4: 前端完善 (2-3 天)
├── 出题界面
├── 进度面板
├── 评估报告
└── 学习计划界面

阶段 5: CLI 工具 (3-4 天)
├── init 命令
├── submit 命令
├── status 命令
└── hint 命令

阶段 6: AI 分析 (2-3 天)
├── 正确性分析
├── 代码风格分析
├── 效率分析
└── 安全性分析

阶段 7: 反馈系统 (1-2 天)
├── 引导式反馈
├── 直接式反馈
└── 混合式反馈

阶段 8: 测试部署 (2-3 天)
├── 单元测试
├── 集成测试
├── 端到端测试
└── Vercel 部署
```

## 三、详细执行计划

### 阶段 1: 数据基础 (1-2 天)

**目标**: 建立数据持久化基础，所有 API 可用

**任务清单**:

- [ ] 1.1 执行数据库迁移
  - 生成 Prisma 迁移
  - 创建数据库表
  - 验证 Schema 正确性

- [ ] 1.2 创建学员管理 API
  - `POST /api/student/create` - 创建学员
  - `GET /api/student/:id` - 获取学员信息
  - `PUT /api/student/:id` - 更新学员信息

- [ ] 1.3 完善聊天 API
  - 集成 Router Agent
  - 集成 Judge 模块
  - 调用对应专业 Agent

- [ ] 1.4 创建评估 API
  - `POST /api/assess` - 水平评估
  - 保存评估结果到数据库

- [ ] 1.5 创建出题 API
  - `POST /api/exam` - 生成题目
  - `POST /api/exam/submit` - 提交答案

- [ ] 1.6 创建规划 API
  - `POST /api/plan` - 生成学习计划

- [ ] 1.7 创建提交 API
  - `POST /api/submit` - 提交代码

**验收标准**:
- 所有 API 可用
- 数据正确存储到数据库
- API 返回正确格式

---

### 阶段 2: 知识基础 (2-3 天)

**目标**: 建立知识库，支持知识检索

**任务清单**:

- [ ] 2.1 收集 OS 教材
  - rCore-Tutorial 文档
  - OS 教科书内容
  - 训练营讲义

- [ ] 2.2 文本分块
  - 将教材分成小段落
  - 保留元数据（来源、章节）

- [ ] 2.3 向量化
  - 使用 OpenAI Embedding 或本地模型

- [ ] 2.4 实现知识检索
  - 语义搜索
  - 关键词搜索
  - 混合搜索

- [ ] 2.5 构建题库
  - 导入现有题目
  - 分类和标签

- [ ] 2.6 知识检索 API
  - `POST /api/knowledge/search` - 知识检索

**验收标准**:
- 知识库包含 500+ 条目
- 语义搜索返回相关结果
- 题库包含 200+ 题目

## 五、风险与应对

| 风险 | 影响 | 应对 |
|------|------|------|
| LLM API 不稳定 | 服务不可用 | 多 Provider 备份 |
| 知识库质量差 | 回答不准确 | 人工审核 + RAG |
| 数据库性能 | 响应慢 | 索引优化 + 缓存 |
| 学员滥用 | 资源浪费 | 个人上限 + 限流 |
