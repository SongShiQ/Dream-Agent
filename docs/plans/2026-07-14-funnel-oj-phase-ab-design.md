# 倒金字塔训练营：导学分流与 OJ Phase A/B 设计

**日期**：2026-07-14  
**状态**：已补充学生体验设计，待确认后实施  
**范围**：导学（约 2,000 人入口）→ 基础 → 专业 → 项目（约 60 人），以及 OJ Phase A/B。  
**前置文档**：[lab-gates-oj-first](./2026-07-14-lab-gates-oj-first.md)。本文件细化其容量、分流和实施契约；不改变“通过 = 自动评测 AC”的原则。

## 1. 目标与边界

训练营不是把 2,000 人都推入同一条高成本实验线，而是用低成本、可解释、可复测的导学流程完成分流；把人工助教和计算资源留给真正进入实验与项目的人。

| 层级 | 规模（设计上限） | 教学目标 | 平台主能力 | 不做的事 |
|---|---:|---|---|---|
| 导学 | 2,000 | 判断起点、补齐先修、识别投入度 | 题库测评、知识卡、短练、学习计划 | 不运行任意代码、不承诺进入项目 |
| 基础 | 约 400–600 | 验证 Rust/工具链/最小 OS 实操 | unit OJ、错误反馈、学习预警 | 不运行 rCore 整仓、不靠静态分过关 |
| 专业 | 约 120–200 | 验证完整 lab 工程能力 | IDE-first、integration OJ、队列 | 不提供网页多文件 IDE |
| 项目 | 约 60 | 完成协作型系统项目 | 项目看板、答疑、老师评分 | 不让助教自动给项目最终分 |

人数是容量预算而非淘汰配额：阶段晋级看达标证据，不能为了凑比例放宽 AC 或跳过门槛。

### 1.1 学生体验北极星

无论处在哪一层，学生打开网页后应在 10 秒内回答四个问题：

1. **我现在在哪里？** 当前阶段、当前关和本周节奏。
2. **下一步只做什么？** 一个可立即开始、可在 15–60 分钟完成的主任务。
3. **做到什么才算前进？** 当前任务的达标证据、剩余条件和解锁结果。
4. **卡住时如何脱困？** 知识卡、公开样例、问答或助教介入，而不是继续盲刷。

产品首页、学习地图、学习计划、练习和 Lab 必须围绕这四个问题协作，不能各自给出相互矛盾的“完成”状态。

## 2. 入口与分流设计

### 2.1 三层证据，不用一次 5 题决定去留

当前 5 题摸底只能给出粗略起点，题目偶然性太高，不能直接决定是否进入实验线。导学改用以下证据：

| 证据 | 首次耗时 | 用途 | 判定方式 |
|---|---:|---|---|
| 诊断卷 | 20–25 分钟，12 题 | 识别 OS、Rust、读代码三个维度 | 固定题库分层抽题；每维至少 3 题 |
| 导学微单元 | 每单元 15–30 分钟 | 检验先修补齐与持续学习 | 知识卡 + 4–6 题小测，允许多次尝试 |
| 工具链自检 | 10–15 分钟 | 验证能否进入基础实验 | 不执行用户代码；提交版本与受控自检结果 |

诊断只做“起点推荐”，不直接淘汰。只有连续完成导学微单元并通过对应小测，才获得基础 OJ 的解锁资格。

### 2.2 推荐路径与晋级规则

```text
注册 → 12 题诊断 → 推荐导学起点
                  ├─ 补弱路径：按薄弱维度完成微单元 → 阶段小测
                  └─ 加速路径：完成挑战卷 + 工具链自检 → 基础候选

导学全链达标 + 环境自检 → 基础 unit OJ
基础必修关 AC + 稳定性检查 → 专业 IDE/integration OJ
专业必修 lab AC + 人工复核材料 → 项目候选
```

默认门槛（作为内容包配置，不硬编码）：

| 关口 | 必要证据 | 反刷量保护 |
|---|---|---|
| 导学 → 基础 | 导学必修单元完成；最近两份小测均 ≥80%；工具链自检成功 | 两次小测覆盖不同题集；不以总答题数替代 |
| 基础 → 专业 | 规定的 `unit_oj` 全部 AC；对应概念小测 ≥80%；最近 7 天至少 2 个学习日 | 必须存在真实 AC，不接受 AI 题或静态分 |
| 专业 → 项目候选 | 规定 `integration_oj` 全部 AC；提交记录完整；一次人工/助教复盘 | 项目名额由老师以协作、时间投入与复盘材料最终确认 |

“加速路径”只减少已掌握的导学内容，不能越过 Rust、工具链或基础 OJ 的 AC 门槛。

### 2.3 运营状态而非简单阶段

保留现有 `currentStage` 作为课程位置，新增/派生三类运营信号：

- `learningStatus`: `active | at_risk | paused | graduated`，由最近活跃、连续失败、完成速度计算。
- `readiness`: 每个维度（theory/rust/tooling/lab）0–100，并保留证据来源。
- `cohort`: 开营批次和课程版本；判题镜像、题库和规则都必须绑定版本。

这样可分别回答“他学到哪里”“是否需要干预”“基于哪个版本被评测”，避免只用一个 stage 字段承载全部含义。

### 2.4 统一“浏览、完成、达标”三种状态

当前体验中，“点开今日三步”“本机勾选任务”和“关卡 AC”容易都被理解为“我完成了”。它们的证据强度不同，必须在数据和 UI 中显式区分：

| 状态 | 含义 | 可信来源 | 能否晋级 | UI 文案示例 |
|---|---|---|---|---|
| `viewed` | 看过内容或打开过入口 | 行为事件，可不跨端保存 | 否 | 已浏览 |
| `personal_done` | 学员完成个人待办 | 服务端待办记录；允许撤销 | 否 | 今日待办已完成 |
| `mastered` | 满足课程规定证据 | 小测达标或 OJ AC | 是 | 已达标 / 已解锁 |

学习地图不能因为当前阶段在后面，就把所有此前阶段显示为“已达标”；最多显示“建议跳过/诊断推荐”，除非存在对应小测或 AC 证据。今日三步也不能因为“点开”就标为完成，应分别显示“已开始”和“标记个人完成”。

第一版不需要为所有浏览行为建复杂表：只需把个人待办完成态改为服务端按 `studentId + date + taskId` 保存；课程达标继续从 `AnswerRecord` 聚合与 `LabGateProgress` 读取。浏览事件可后置。

### 2.5 学员首页：统一的“我的状态与下一步”

导学、基础、专业、项目使用同一首页框架，但主卡片按阶段改变：

| 区块 | 学员看到的内容 | 数据来源 | 行为 |
|---|---|---|---|
| 当前状态 | 阶段、cohort、学习状态、距下一关的条件 | `Student` + 达标聚合 | 查看完整路径 |
| 唯一主任务 | “完成 Rust 所有权小测 4/6”或“修复 rust-ownership 的 CE” | 规则引擎 | 一键进入正确页面 |
| 达标清单 | 已具备 / 缺失的条件及每项证据 | 测评、OJ、工具链自检 | 不允许伪造勾选 |
| 卡住了怎么办 | 依据失败类型给出知识卡、公开示例、问答/求助 | 错题与 JudgeRun | 进入补救流程 |
| 本周节奏 | 计划工作量、连续学习日、截至时间（若有） | 服务端任务与 cohort | 调整个人计划 |

这不是再增加一个入口，而是替代“要在聊天、地图、练习、实验中猜下一步”的导航负担。导学主卡默认指向微单元，基础指向最近未 AC 的 unit OJ，专业指向最近解锁的 IDE-first Lab，项目指向老师定义的里程碑。

## 3. 内容、反馈与受挫干预策略

1. **导学规模优先稳定与自助。** 固定题库、知识卡、错题归因、模板化计划是默认服务；LLM 仅解释与变式练习，不能写入晋级证据。
2. **基础阶段优先可验证。** 每题只考一个 Rust/OS 微技能，隐藏测试验证边界；反馈输出错误类别和下一步，不泄露隐藏断言。
3. **专业阶段优先真实工程。** 学员在 IDE 本地工作，提交 patch；网页是任务板、日志和答疑入口。
4. **项目阶段优先协作。** 展示里程碑、风险和复盘模板；老师负责验收与最终评价。

人工干预按风险排序：连续两次导学小测失败、基础题连续 3 次 WA、同一专业关超过 7 天 PENDING/WA，进入助教待跟进列表。只有这类小比例用户进入人工队列。

### 3.1 导学微单元的页面契约

导学不是“题库列表”。每个单元固定为“目标 → 阅读 → 公开示例 → 小测 → 结果与下一步”的 15–30 分钟闭环，并显示：

- 本单元解决的具体能力，例如“能解释所有权移动为何阻止二次使用”。
- 预计耗时、完成条件、与后续实验的关系。
- 小测已答/正确数、是否已达标、是否需要换题集重测。
- 未达标时的最短补救路径，而非只显示分数。

12 题诊断仅决定推荐起点；完成微单元和小测才产生 `mastered` 证据。加速学生看到的是“已由诊断建议跳过，仍可挑战验证”，不是虚假的“已完成”。

### 3.2 基础与专业的提交反馈契约

| 场景 | 必须展示 | 不应展示/承诺 |
|---|---|---|
| unit OJ 提交前 | 允许提交的文件、公开测试、本地命令、资源限制 | 可上传整个 crate 或任意命令 |
| unit OJ 排队中 | `PENDING`、队列位置/预计等待、可安全离开页面 | 长时间无解释的加载动画 |
| unit OJ 失败 | WA/CE/RE/TLE 类别、截断公共日志、下一步补救 | 隐藏测试断言与完整内部环境 |
| integration OJ | 基线仓库/版本、patch 范围、CLI 命令、最近提交时间线 | 网页多文件 IDE 或静态分可过关 |
| 项目 | 候选资格、老师里程碑、协作与复盘模板 | “平台自动判定项目通过” |

连续 3 次非 AC 时进入“诊断模式”：先显示最相关的知识卡和公开样例，再给一次仅跑公开测试的自检入口，最后提供带上下文的问答/助教求助。门槛不降低，帮助路径必须变短。

### 3.3 跨端与可信进度

个人计划若只在浏览器本地勾选，会导致学生换电脑后误以为数据丢失；训练营证据也无法用于运营。需要：

- 日计划完成态服务端保存，并按日/批次可回溯；它仍只是 `personal_done`。
- 小测、AC、工具链自检保留服务端证据，并能重建 `mastered` 状态。
- UI 不把“已点开”渲染为“已完成”，也不把个人待办渲染为“已达标”。
- 学生可看到自己的提交时间线；助教看到同一证据的聚合视图。

## 4. Phase A：数据与 OJ 纪律（实施设计）

### 4.1 目标与验收

Phase A 的目标不是“已经有 OJ”，而是让任何未来判题器都只能通过同一条受审计的通道写入 AC 和进度。

验收条件：

1. 一次 schema migration 可在空库和当前开发库执行。
2. 客户端提交无论携带何种 `isPassed`，均不能使关卡通过。
3. 只有受服务端保护的判题完成函数在 `verdict=AC` 时能写入 `LabGateProgress.passed`。
4. 每条提交都记录 `gateId`、`judgeKind`、verdict、截断日志、内容/版本信息。
5. Lab API 在数据库未迁移时给出明确维护错误，而不是 500。
6. 学习地图和今日计划能区分 `personal_done` 与真实 `mastered`，换设备后个人完成态不丢失。

### 4.2 数据模型调整

现有未提交 schema 已加入 `LabGateProgress` 和 submission verdict，但缺少迁移，且还不足以支持异步判题。Phase A 应一次性补齐最小可演进模型：

| 模型/字段 | 责任 | 说明 |
|---|---|---|
| `CodeSubmission` | 学员提交原件 | 增加 `contentVersion`、`sourceKind(web/cli)`；`code`/patch 受长度上限保护 |
| `JudgeJob`（新增） | 队列与重试 | `queued/running/finished/failed`、attempt、leaseUntil、judgeVersion、资源限制快照 |
| `JudgeRun`（新增或与 Job 合并） | 可审计的单次运行 | exitCode、durationMs、memory、publicLog、internalLogRef、verdict |
| `LabGateProgress` | 派生进度 | 从历史 AC 更新；不让 UI 直接写；保存 `contentVersion` |
| `Student` | 漏斗运营 | `cohort`、`learningStatus`、最近活跃时间；readiness 可先以 JSON/独立表二选一 |
| `DailyTaskProgress`（新增，轻量） | 跨端个人计划 | `studentId + date + taskId` 唯一；只记录 `personal_done`，不参与晋级 |

`CodeSubmission.verdict` 在提交时应为 `PENDING`（将来只有完全不进入判题队列的旧静态分析才使用 `STATIC`）。这样 UI 和 API 先使用最终状态机，不必在 Phase B 返工。

状态机：

```text
submit → PENDING → JudgeJob.queued → running → AC | WA | CE | RE | TLE | SE
                                      └→ retry（仅系统性 SE，且次数受限）
AC → 单事务：submission.isPassed=true + LabGateProgress=passed + 解锁后继关
```

### 4.3 API 与权限边界

- `POST /api/submit`：校验 student/gate/version/长度，创建 submission + job；不接受 verdict、judgeLog、isPassed。
- `POST /internal/judge/complete`：仅 worker 身份可调用，校验 job lease 与幂等键；完成事务。
- `GET /api/submissions/:id`：只返回安全的公共日志和状态；前端轮询使用指数退避。
- `GET /api/labs`：只读 dashboard；`POST /api/labs` 仅允许 sync/read，不提供 passed 操作。

开发期可用进程内 worker，但 `/internal` 接口与鉴权契约仍应先建好，避免生产时出现客户端可伪造 AC 的路径。

### 4.4 Phase A 实施顺序

1. 先备份 `prisma/dev.db`，生成并审阅 migration；不修改已有 migration。
2. 生成 Prisma client，跑 typecheck 与现有测试。
3. 创建 Job/Run 模型、服务层和状态机单元测试；替换 submit API 的“直接 STATIC”为创建 PENDING job。
4. 给 gates 加 `contentVersion`、`judgeSpec` 占位契约；校验 JSON。
5. 更新 LabPanel/CLI：展示队列状态、最近 verdict 和不可伪造的 pass 状态。
6. 将日计划完成态迁到服务端；地图的“今日三步”改为“已开始 / 个人完成 / 已达标”的诚实文案。
7. 加 API 集成测试：伪造 `isPassed`、重复 complete、非 AC complete、并发 AC 都不能破坏状态。

Phase A 完成后仍不声称可判题；它只保证“所有提交被可靠排队，且未来 AC 可安全地更新进度”。

## 5. Phase B：基础 unit OJ（实施设计）

### 5.1 首批题目与范围

先实现 5 个小题闭环，不直接接整套 rustlings：

| Gate | 技能 | 输入 | 判定 |
|---|---|---|---|
| `env-check` | 工具链基础 | 受控环境指纹，不执行任意脚本 | 格式/版本规则 |
| `rust-variables` | 变量与 mut | `lib.rs` 单文件 | 公开 + 隐藏 cargo test |
| `rust-ownership` | move/borrow | `lib.rs` 单文件 | 编译 + 测试 |
| `rust-result` | Result/错误处理 | `lib.rs` 单文件 | 边界值测试 |
| `basic-syscall-model` | 系统调用抽象 | 小 crate | 单元测试 |

`lab1-batch` 留在 Phase C：它是仓库级问题，不应为了尽快上线而伪装成 unit OJ。

### 5.2 题包契约

每题放在版本化目录 `data/judges/unit/<gateId>/<version>/`：

```text
template/              # 固定 Cargo.toml、lib.rs 占位
tests/public.rs         # 学员可见的最小测试
tests/hidden.rs         # 仅评测镜像内
judge.json              # image、command、timeout、memory、最大源码大小
README.md               # 目标、输入输出/接口、提交说明
```

学员只能替换声明的源码文件，不能上传 Cargo.toml、build.rs、依赖锁文件或任意 shell 命令。如此既稳定，又避免 Cargo dependency 和 build script 成为攻击入口。

### 5.3 执行安全与容量

生产使用每次提交一个 Docker 容器：非 root、只读 rootfs、无网络、只挂载临时工作目录、CPU/内存/PID/磁盘/总时间限制。默认建议：编译+测试 20 秒、1 CPU、512 MiB、64 MiB 临时盘、64 个 PID、源码 64 KiB、日志 32 KiB。

容量先按导学 2,000 人中 25% 在基础期、20% 峰时同时提交估算，即约 100 个提交/分钟的突发，而非常态。设计为：

- Web API 只入队，不能等待 `cargo test`。
- 单 worker 初始并发 4；达到阈值后返回排队位置与预计等待。
- 以每次 5–10 秒估算，4 并发能处理 24–48 提交/分钟；开营峰值需 3–5 个 worker 或限流/预约窗口。
- 每位学员每题 60 秒内最多 1 次、每小时最多 12 次；CE 反馈立即返回但仍计入限流。
- 队列长度、等待时间、AC/WA/CE/TLE、容器失败率必须进入运营仪表盘。

先用单机 Docker 做小规模内测；正式承载 2,000 人前必须进行压测并把 worker 部署到 Linux。Windows Docker Desktop 仅用于本地开发验证。

### 5.4 反馈与异常策略

| 结果 | 面向学员 | 后台动作 |
|---|---|---|
| AC | 已通过、解锁下一关、推荐下一步 | 原子更新进度 |
| WA | 指出失败类别/公开用例，不给隐藏断言 | 记录概念标签与尝试次数 |
| CE | 编译诊断（截断、脱敏） | 推荐知识卡/公开测试 |
| TLE/RE | 资源或运行时类别 | 保留完整内部日志，学员见摘要 |
| SE | 明确“系统异常，不计失败” | 有限自动重试，告警值班 |

连续 3 次非 AC 不自动降低题目难度，而是给“诊断模式”：公开样例、相关知识卡和助教问答入口；这样既保持 gate 的刚性，也避免大规模入口用户被静默卡死。

### 5.5 Phase B 验收与灰度

1. 本地：错解→WA、语法错→CE、死循环→TLE、正确解→AC；每种都有自动测试。
2. 安全：容器无法联网、无法写宿主、不能以 root 运行；超限稳定终止。
3. 一致性：重复完成 job 不重复解锁；同一 AC 的历史记录可重建进度。
4. 容量：按目标峰值压测，p95 排队+执行时间、失败率达到上线阈值后才扩大。
5. 灰度：先 30 名内测 → 200 名基础组 → 全量；每档至少观察一周数据。

## 6. 决策与下一步

本方案建议的实施优先级：

1. **补齐并验证 Phase A migration、状态机和跨端个人进度**（当前代码的硬缺口）。
2. **先设计并落地“我的状态与下一步”主卡**，让现有地图、计划、练习与 Lab 使用同一份达标聚合。
3. **改造 12 题诊断与导学微单元/小测规则**，先让 2,000 人入口有可靠分流和补救闭环。
4. **搭建 Phase B 的 5 题 unit OJ 内测闭环**，先用 Docker/Linux worker 验证安全和容量。
5. 依据基础期真实数据，才启动 Phase C 的 rCore integration OJ。

需在开工前确认的产品参数：导学必修单元清单、基础候选人数/节奏、项目候选的人工复核规则、正式环境是否可提供 Linux Docker worker，以及 cohort 的开营/截止节奏。未确认这些参数前，不能把“2000→60”硬写成自动淘汰逻辑。
